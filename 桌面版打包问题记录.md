# 桌面版打包问题记录

记录日期：2026-06-23

## 问题现象

浏览器开发版导入IQC数据后可以显示：

- 内部加工（一楼自制）专项分析
- 特采风险专项分析

已打包的桌面程序导入数据后未显示上述两个板块。

## 核查结论

### 1. 专项分析代码没有遗漏

当前网页构建文件和桌面目录中的JavaScript文件哈希完全一致，并且桌面构建文件中可以检索到：

- `内部加工（一楼自制）专项分析`
- `特采风险专项分析`
- `internalModes`
- `specialAnalysis`

因此，问题不是React页面或分析引擎在打包时被删除。

### 2. release目录存在两个不同的发布版本

当前`release`目录同时存在：

- `QMS质量分析平台-免安装版`
- `QMS璐ㄩ噺鍒嗘瀽骞冲彴-鍏嶅畨瑁呯増`

产生原因：

- `scripts/package-portable.ps1`使用UTF-8中文字符串。
- Windows PowerShell 5按错误编码读取无BOM的UTF-8脚本。
- 新构建被复制到乱码名称目录和乱码名称压缩包。
- 正常中文名称的目录/压缩包可能仍是较早发布物。

这会导致使用者看似打开了“最新桌面版”，实际运行的是另一个旧目录或旧压缩包。

### 3. 桌面版数据存储尚未完成大数据量验收

桌面版通过`file://`加载页面，IndexedDB位置为：

`%APPDATA%\quality-analytics-web\IndexedDB\file__0.indexeddb.leveldb`

当前本机检查到该数据库日志体积很小，没有证据证明完整IQC、IPQC、OQC和DQA数据已经成功持久化。

两个专项板块的显示条件是：

- `data.iqc.internalModes`存在时显示内部加工专项。
- `data.iqc.specialAnalysis`存在时显示特采专项。

如果桌面版导入后仍停留在内置样例数据，或大型解析数据未成功写入/恢复，这两个字段不存在，板块就不会渲染。浏览器开发版使用`http://127.0.0.1`存储空间，与桌面版的`file://`存储空间完全独立，因此浏览器能显示不能证明桌面版已经保存和恢复成功。

## 最可能的原因排序

1. 实际打开了正常中文名称下的旧发布物，而最新发布物被生成到了乱码目录。
2. 桌面版导入的大型解析数据没有成功持久化，重启后回到了内置样例数据。
3. 用户在浏览器版已经导入的数据不会自动同步到桌面版，需要桌面版单独导入。

## 后续最终打包前必须修复

1. 打包脚本不得依赖PowerShell 5解析中文路径：
   - 脚本保存为带BOM的UTF-8；或
   - 发布目录、临时目录全部使用ASCII名称，最后再统一重命名。
2. 每次打包前清空`release`中所有旧目录、乱码目录和旧ZIP，避免多版本混用。
3. 在页面和桌面窗口中显示唯一版本号、构建日期和构建哈希。
4. 桌面版改用Electron主进程管理持久化文件或SQLite，不继续把几十万行解析数据完全依赖`file:// IndexedDB`。
5. 保存流程必须显示明确状态：
   - 正在保存
   - 保存成功
   - 保存失败及失败原因
6. 启动恢复后显示：
   - 已恢复文件数量
   - 各模块行数
   - 是否已生成`internalModes`和`specialAnalysis`
7. 最终验收必须在免安装版内实际导入两份IQC文件，确认：
   - 外部供应商总体趋势出现
   - 内部加工专项出现
   - 特采专项出现
   - 关闭程序重新打开后仍然出现
   - 同名文件替换和删除正常

## 当前处理决定

- 暂不继续打包。
- 保留当前浏览器开发版继续迭代。
- 项目开发完成后，先完成桌面持久化改造和上述验收，再生成唯一最终发布包。
